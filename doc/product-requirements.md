# プロダクト要求定義書

室内装置予約システム PoC に関する要求の SSOT。実装の可否判断やスコープ変更時に更新する。

---

## 1. プロダクトビジョンと目的

- **ビジョン**: 研究室内の共有装置を、認証済みユーザーが時間帯単位で予約・確認できるようにする。
- **目的**: PoC として、JWT 連携したバックエンド API と最小限のフロントで「装置の検索・予約・管理」が成立することを示す。

## 2. ターゲットユーザーと課題・ニーズ

| ユーザー | 課題・ニーズ |
|----------|----------------|
| 研究室メンバー | 装置の空き状況を把握し、衝突なく予約したい |
| 管理者 | 装置マスタの登録・更新・削除を行いたい |

## 3. 主要な機能一覧

- Keycloak によるログイン（JWT）
- 装置の一覧・詳細・CRUD（一般閲覧と管理者更新の区別）
- 装置のファセット／インクリメンタル検索
- ログインユーザーに紐づく予約の一覧・作成・更新・削除（同一装置の時間重複禁止）
- ユーザー情報の参照（自己・管理者による一覧／詳細）

## 4. 成功の定義（PoC）

- ローカル環境で Keycloak・PostgreSQL・API・フロントが起動し、エンドツーエンドで予約が一通りできること。
- バックエンドに対する自動テストがモックなしで通り、カバレッジ目標に近い水準であること（詳細は [@doc/development-guidelines.md](development-guidelines.md)）。

## 5. ビジネス要件（PoC 範囲）

- マルチテナントや課金は対象外。
- 監査ログ・承認フローは必須としない（将来拡張）。

## 6. ユーザーストーリー（例）

- 研究室メンバーとして、装置を条件で絞り込みたいので、カテゴリや場所で検索したい。
- 研究室メンバーとして、利用したい時間帯を予約し、自分の予約一覧で確認したい。
- 管理者として、新しい装置を登録し、状態（利用可・保守中等）を更新したい。

## 7. 受け入れ条件（抜粋）

- 未認証で閲覧可能な装置 API と、認証必須の予約 API の区別が仕様どおりであること。
- 予約の時間帯が同一装置上で重なる場合、作成・更新が拒否されること（`cancelled` を除くルールは [@doc/functional-design.md](functional-design.md)）。
- 装置に紐づく予約がある場合、装置削除が整合性ルールに従うこと。
